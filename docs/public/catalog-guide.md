# Loregist Catalog 기능 사용 가이드

## 개요

Loregist의 catalog 기능은 위키 문서에서 topic(도메인 개념)과 decision(아키텍처·정책 결정)을 추출해 구조화된 카탈로그를 구성한다. 이 가이드는 wiki 소스 선정부터 섹션 단위 분석 분기까지 catalog 기능을 올바르게 활용하는 방법을 설명한다.

---

## 1. wiki 소스 선정 기준

### 포함 추천 기준

다음 특성을 가진 문서는 catalog 분석 대상으로 적합하다.

- **최종 갱신 헤더가 있는 문서**: `## 최종 갱신`, `## Last Updated` 등의 헤더가 존재하는 문서는 정기적으로 관리되고 있음을 나타낸다. 최신 상태가 유지될 가능성이 높아 추출 품질이 좋다.
- **구조화된 표·섹션이 있는 문서**: 마크다운 표, 헤더 계층(`##`, `###`), 목록 구조가 명확한 문서. 섹션 단위 분석이 용이하고 topic/decision 경계를 잘 잡아낸다.
- **토폴로지·정책·결정 이력 성격의 문서**: 시스템 구성도, 인프라 토폴로지 설명, 정책 문서, 아키텍처 결정 기록(ADR). catalog의 핵심 가치를 담고 있는 문서 유형이다.

### 제외 추천 기준

다음 특성을 가진 문서는 catalog 분석에서 제외하는 것을 권장한다.

- **연락처·IP 목록**: 개인 연락처, 서버 IP 목록, 자격증명 목록 등은 topic이나 decision으로 추출할 내용이 없다. 주소록 형태의 문서는 분석 노이즈만 늘린다.
- **날짜별 로그**: 일별 배포 로그, 날짜별 변경 이력 등 시간 순으로 나열된 이벤트 기록. 반복적인 도메인 개념이나 명시적 결정이 포함되지 않는 경우가 대부분이다.
- **임시 메모**: 작성 중인 초안, 개인 메모, 브레인스토밍 문서. 구조가 정제되지 않아 분석 결과의 품질이 낮다.

## 2. topic 해당 예시 vs 비해당 예시

topic은 위키 문서에서 **반복 등장하는 도메인 개념·기술 영역**을 가리킨다. 단발성 사건이 아니라, 팀이 지속적으로 참조하는 개념적 단위다.

### 해당 예시

| 예시 | 설명 |
|------|------|
| **DB 구분 (서비스 DB vs 원천 DB)** | 서비스가 사용하는 DB와 데이터 원천 DB를 구분하는 개념. 여러 섹션에 걸쳐 반복 등장하며 아키텍처 전체에 영향을 준다. |
| **인프라 토폴로지** | 클러스터 구성, 네트워크 존 분리, 서비스 배치 구조 등 시스템의 물리적·논리적 배치 개념. |
| **기술 스택 구성** | 데이터 파이프라인 구성 요소(NiFi, Kafka, PDIS 등), 서비스 프레임워크 조합 등 반복 참조되는 기술 집합. |
| **배포 환경 구분 (개발/스테이징/운영)** | 환경별 정책 차이나 분기 기준으로 계속 등장하는 개념. |

### 비해당 예시

| 예시 | 설명 |
|------|------|
| **날짜별 회의록** | 특정 날짜에 일어난 논의 기록. 반복되는 개념이 아니라 1회성 이벤트다. |
| **1회성 이벤트** | 특정 장애 발생, 특정 날 배포 등 반복되지 않는 단건 사건. |
| **연락처 목록** | 담당자 이름·이메일·전화번호 목록. 도메인 개념이 아니라 운영 정보다. |
| **단순 설정값 나열** | 환경 변수 값, 포트 번호 목록 등 의미 해석 없이 나열된 설정 데이터. |

## 3. decision 해당 예시 vs 비해당 예시

decision은 위키 문서에서 **명시적 선택이 이루어진 아키텍처·정책 결정**을 가리킨다. "왜 이렇게 했는가"에 해당하는 선택 행위가 기록된 것이어야 한다.

### 해당 예시

| 예시 | 설명 |
|------|------|
| **NiFi → PDIS 방화벽 요청 경로 결정** | NiFi에서 PDIS로 데이터를 보낼 때 방화벽 통과 경로를 특정 방식으로 결정했다는 기록. 대안 비교 또는 배경이 함께 있으면 더 명확하다. |
| **배포 전략 선택 (블루/그린 vs 롤링)** | 두 가지 이상의 방안 중 하나를 선택했다는 명시적 결정. 이유나 트레이드오프 설명이 포함된다. |
| **DB 분리 결정 (서비스 DB와 원천 DB 분리)** | 단일 DB 사용 대신 역할별 DB를 분리하기로 결정했다는 기록. |
| **인증 방식 선택 (JWT vs 세션)** | 보안·성능 등의 기준으로 인증 방식을 명시적으로 선택한 결정. |

### 비해당 예시

| 예시 | 설명 |
|------|------|
| **일반 현황 설명** | "현재 A 시스템은 B를 사용하고 있다"는 식의 현황 기술. 선택 행위가 없다. |
| **미결정 사항 나열** | "방안 A와 B를 검토 중"처럼 아직 결정되지 않은 사항. 결정이 내려지지 않았으므로 decision이 아니다. |
| **단순 설정값 목록** | 타임아웃 값, 최대 연결 수 등 구성 파라미터 나열. 결정 맥락 없이 값만 있는 경우. |
| **역할 설명** | "A 팀은 B를 담당한다"는 조직 설명. 선택이 아니라 역할 기술이다. |

## 4. 섹션 단위 분석 분기 기준

### 섹션 분할 방식

Loregist catalog 분석은 `##` 헤더를 기준으로 문서를 섹션 단위로 분할한다. 각 섹션은 독립적으로 topic 또는 decision 후보로 평가된다.

```
# 문서 제목            ← 문서 메타 (분석 제외)
## 섹션 A              ← 분석 단위 1
### 하위 섹션 A-1      ← 섹션 A 내 맥락으로 처리
## 섹션 B              ← 분석 단위 2
```

`###` 이하 하위 헤더는 해당 `##` 섹션의 내용으로 통합해 분석한다.

### topic 판별 신호

다음 신호가 섹션 제목이나 본문에 나타나면 topic 후보로 분류한다.

- **섹션 제목 키워드**: "개념", "구조", "구성", "현황", "아키텍처", "개요"
- **명사 중심 섹션 제목**: 동사 없이 명사구로만 이루어진 제목 (예: `## 데이터 파이프라인 구성`, `## 인프라 토폴로지`)
- **본문 패턴**: 표나 목록으로 구성 요소를 열거, 다이어그램 설명, 역할별 비교

**예시**

```markdown
## 서비스 DB 구성

| DB 명 | 역할 | 엔진 |
|-------|------|------|
| svc_db | 서비스 읽기/쓰기 | PostgreSQL |
| src_db | 원천 데이터 적재 | MySQL |
```

→ 명사 중심 제목 + 표 구조 → **topic** 신호

### decision 판별 신호

다음 신호가 섹션 제목이나 본문에 나타나면 decision 후보로 분류한다.

- **섹션 제목 키워드**: "결정", "선택", "정책", "기준", "방침", "채택", "이유", "배경"
- **본문 패턴**: "방안 A / 방안 B → A 선택", "~하기로 결정", "~을 채택한 이유", "트레이드오프"
- **비교 구조**: 두 가지 이상 대안을 나열한 후 하나를 선택하는 문장 흐름

**예시**

```markdown
## NiFi-PDIS 연동 방화벽 경로 결정

방안 A: NiFi → DMZ → PDIS (직접 연결)
방안 B: NiFi → 중계 서버 → PDIS

보안 정책상 DMZ 직접 연결이 불가하여 방안 B를 채택하기로 결정했다.
```

→ "결정" 키워드 + "방안 A/B → B 채택" 패턴 → **decision** 신호

### 분기 판별 흐름 요약

```
섹션 진입
  │
  ├─ 제목에 "결정/선택/정책/기준/방침" 포함?
  │    YES → decision 후보
  │
  ├─ 본문에 "방안 X → Y 선택", "~하기로 결정" 패턴 존재?
  │    YES → decision 후보
  │
  ├─ 제목이 명사구 중심이고 표·목록으로 구성 설명?
  │    YES → topic 후보
  │
  ├─ 본문에 도메인 개념이 반복 등장하고 설명 구조?
  │    YES → topic 후보
  │
  └─ 위 모두 해당 없음 → 추출 제외 (연락처·로그·단순 나열 등)
```

### 주의 사항

- 하나의 섹션이 topic과 decision 신호를 동시에 포함할 수 있다. 이 경우 **주된 서술 목적**을 기준으로 분류하거나 양쪽 모두 추출한다.
- 섹션 본문이 짧거나(3줄 미만) 정보가 빈약한 경우 추출 품질이 낮으므로 제외를 고려한다.
- `## 최종 갱신`, `## 참고`, `## 부록` 등 메타 섹션은 topic/decision 분석에서 제외한다.

## 5. edges frontmatter (카탈로그 그래프 링크)

`_wiki/T-xxx.md`·`D-xxx.md` 의 frontmatter는 항목 간 그래프 연결을 `edges:` 필드로 표현한다.

```yaml
---
id: D-007
related: []                       # 소스 파일 경로 (plans/done/... 등 외부 문서)
edges: [T-010, T-012, D-008]      # 내부 카탈로그 id 전용 (T-xxx / D-xxx)
---
```

- **`edges:` = 내부 카탈로그 id 리스트**(`T-xxx`/`D-xxx`)다. 항목 간 의미적 연관을 그래프로 잇는다.
- **`related:` 와 구분**한다 — `related:` 는 증류 출처가 된 **소스 파일 경로**(예: `plans/done/...md`)이고, `edges:` 는 **카탈로그 내부 id**다. 둘은 의미가 달라 별도 필드로 분리돼 있다.
- **양방향 대칭 규칙**: A의 `edges:` 에 B가 있으면 B의 `edges:` 에도 A가 있어야 한다(A→B 존재 시 B→A 필수). 비대칭은 lint `asymmetric` 위반이다(섹션 6).
- **렌더**: `TOPICS.md`·`DECISIONS.md` 표의 **"연결" 컬럼**이 각 항목의 `edges:` 를 렌더한다. related 컬럼(`TOPICS.md`는 "관련 파일", `DECISIONS.md`는 "related")과는 별개 컬럼이다.

## 6. lint_edges 검사 규칙

`loregist catalog --lint` 는 `_wiki/` 전체의 `edges:` 무결성을 검사한다(`lint_edges()`). 4종 규칙을 severity로 구분하며 `(error_count, warning_count)` 를 반환한다.

| 규칙 | severity | 설명 |
|---|---|---|
| `dangling` | error | `edges:` 의 id가 실존하는 카탈로그 id 집합에 없음(없는 노드 참조) |
| `asymmetric` | error | A→B 엣지가 있으나 B→A 역엣지가 없음(비대칭) |
| `self-ref` | error | 자기 id를 자신의 `edges:` 에 포함(자기참조) |
| `orphan` | warning | `edges:` 가 비어 있고 다른 항목으로부터 피참조도 없음(고립 노드) |

- **error**(dangling·asymmetric·self-ref)는 `error_count` 에, **warning**(orphan)은 `warning_count` 에 집계된다.
- `error_count > 0` 이면 `catalog --lint` 가 비정상 종료(exit ≠ 0)하여 CI·발행 게이트에서 차단할 수 있다. orphan은 warning이라 종료 코드에 영향을 주지 않는다.
