# docs/ 화이트리스트화 (기본 차단)

> 소스: `plans/done/2026-06-19.stashdex_내부메모_경로이전_docs화이트리스트_계획.md §설계 결정: 블랙리스트 → 화이트리스트`

dev → public 차단이 경로 기반에서 `^plans/` 단일 패턴뿐이라 `docs/`는 구조적 보호가 전혀 없었고, 현재 public에 docs가 없는 것도 audit 내용 스캔에 트리거 단어가 우연히 걸린 결과일 뿐이었다. 이는 블랙리스트의 본질적 실패 모드로, 트리거 단어가 없는 미래의 내부 docs 문서가 내용 스캔을 통과해 그대로 공개 유출될 위험이 있다. 이를 역전하기 위해 `docs/` 전체를 기본 비공개로 막고 공개할 문서만 명시 허용 경로(`docs/public/`)에 두는 화이트리스트로 전환한다. 새 내부 문서는 기본 차단(안전)되고 공개하려면 `docs/public/`에 의도적으로 넣어야 하는(opt-in) 구조라, 프로젝트의 "유출 0" 철학과 정합한다. 화이트리스트 단위는 자기설명적이고 구현이 단순한 서브트리(`docs/public/`)를 채택했고(파일 allowlist는 유지보수 부담으로 미채택), 차단 규칙은 `^plans/` 또는 (`^docs/` AND NOT `^docs/public/`)으로 publish.sh·audit.sh·public .gitignore 3곳에 동일 반영한다.
