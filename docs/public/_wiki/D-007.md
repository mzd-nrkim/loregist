# OKF frontmatter 차용 (catalog 한정)

> 소스: `plans/done/2026-06-15.01.OKF_적용_검토.md §차용할 것`

OKF(Open Knowledge Format)를 검토한 결과 통째 적용은 부적합하다고 판단하고, catalog 레이어에 한해 두 가지만 차용하기로 했다. OKF는 "파일로 남겨 다 보이게"인 반면 loregist의 cold 레이어는 "파일을 DB로 밀어넣어 안 보이게"라 정반대이고, 벤더 중립·BigQuery 인제스트·웹 enrichment·포터빌리티는 개인·단일 사용자·LLM 소비 환경에서 가치가 거의 없기 때문이다. 차용할 것은 ① concept/decision 하나당 파일 하나의 일관된 YAML frontmatter 스키마(`id/type/date/status/tags/related`) — 자동 추출이 평문 헤딩 파싱 대신 정규 frontmatter 파싱으로 견고해지고, ② `[[link]]`를 catalog의 1급 엣지(`related`)로 승격해 의사결정 이력을 그래프로 만드는 것이다. Visualizer(HTML 그래프 뷰어)·Enrichment Agent 웹 pass·번들 포터빌리티는 차용하지 않는다. 결론적으로 고볼륨 원시 기록(로그)은 loregist 검색이, 소수 정제 지식(decision/topic)은 OKF식 catalog가 맡는 파이프라인의 두 끝으로 분업한다.
