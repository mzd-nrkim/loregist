# catalog edges/related 필드 분리

> 소스: `plans/done/2026-06-23.catalog_related_그래프_엣지_보강.md §related:를 쓰지 않고 edges:를 신설하는 이유`

catalog T/D 파일에서 `related:`와 `edges:` 두 필드를 의미에 따라 분리하기로 결정했다.

**문제**: 기존 `related:` 필드는 catalog-update 스킬의 B-5(중복 검사 시 소스 파일 경로 append)와 D-007이 차용한 OKF 그래프 링크(내부 T/D id 참조)가 동일 필드를 사용해 의미가 충돌했다. `related: [D-008, plans/foo.md]`처럼 catalog id와 파일 경로가 혼합되면 lint 단계에서 파일 경로를 "존재하지 않는 catalog id"로 오탐하는 구조적 문제가 발생한다.

**해결**: 두 신호를 별도 필드로 분리한다.
- `related:` — catalog-update B-5 전용. 외부 소스 파일 경로(예: `plans/done/foo.md`, `etc/infra.md`)를 append하는 스킬 영역.
- `edges:` — 내부 항목 간 그래프 링크 전용. catalog id(`T-xxx`/`D-xxx`) 참조만 허용. `stashdex catalog --lint`가 dangling/asymmetric/self-ref/orphan 오류를 이 필드 기준으로 검사한다.

**구현**: 45개 T/D 파일에 `edges: []` 필드를 신설하고 양방향 대칭 엣지를 채웠다. `catalog_gen.py`에 `--lint` 플래그를 추가해 edges 무결성을 자동 검증한다(`--json`으로 구조화 출력 가능). TOPICS.md·DECISIONS.md 인덱스 테이블에 "연결" 컬럼을 추가해 edges를 시각화한다.
