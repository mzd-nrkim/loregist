## 프로젝트 해석

1. `--project <key>` 인자가 있으면 그 프로젝트 사용
2. 없으면 `stashdex project current` (cwd 기준 자동 추론)
3. 추론 실패 시 사용자에게 프로젝트 키 질문

`docs_root`: 추론된 프로젝트의 docs_root 값 (`stashdex project list`로 확인)

→ loregist/CLAUDE.md "스킬 공통 — 프로젝트 추론 규칙" 적용

이 스킬의 모든 상대 경로는 `{docs_root}/` 기준이다.

# 역할

VM 작업 히스토리(로그 파일)를 주제별 문서에 기입 → 분석 → 다음 할 일 제안 + 명령어 제공.

# 입력 분석

1. **로그 파일 경로** — 상대경로면 `{docs_root}/{오늘날짜}/` 기준. 절대경로도 허용.
2. **옵션**: `--topic=<경로|주제명>` (기입 대상 주제별 문서 명시), `--doc=<경로>` (인덱스 명시)

# 기입 대상 탐색

우선순위: `--topic` > `--doc` > **자동 매칭** (오늘 인덱스의 `> 상세:` 링크에서 로그 내용 키워드로 매칭, 1개면 자동 선택, 0/2+이면 사용자 질문) > 인덱스 최하단 (레거시)

# 처리 절차

## 1단계: 로그 파일 읽기

파일이 없으면 에러 후 중단.

## 2단계: 히스토리 기입

주제별 문서의 `## 히스토리` 섹션 최하단에 엔트리를 추가한다 (섹션 없으면 생성).

### 히스토리 섹션 구조

`## 히스토리`는 시간순 `### {HH:MM} — {제목}` 엔트리의 나열이다. 엔트리 외 다른 내용을 포함하지 않는다.

- `###` 헤딩은 `### {HH:MM} — {제목}` 형식만 허용 — 비타임스탬프 서브섹션(`### 참조자료`, `### 스크립트 반입` 등) 생성 금지
- 각 엔트리는 자기완결적 — "위 참조" 등 엔트리 외부 참조 금지. 필요한 코드블록은 분량에 무관하게 엔트리 내에 직접 포함
- 기존 엔트리 삭제·수정 금지

> 새 히스토리는 반드시 이 구조로 작성한다. doc-filter는 레거시 문서의 비타임스탬프 `###`를 보존적으로 귀속 처리하지만, 신규 작성 시 비표준 `###`에 의존하지 않는다.

### 엔트리 템플릿

```markdown
### {HH:MM} — {히스토리 요약 제목}

**실행 내용:** {1줄 요약}

**결과:**
- {주요 결과 bullet}

**판단:**
- {결과 해석}

**다음 할 일:**
- {구체적 다음 행동}

**실행 명령어:**
{코드블록 — 분량 무관, base64 분할·다단계 명령어 모두 포함. VM 환경 규칙에 맞게 조정 (단일 코드블록, 복잡한 파일 전송 시 SCP 분리)}
```

### 기입 규칙

- 시각은 로그 파일명에서 추출 (예: `1738.log` → `17:38`)
- 하나의 로그에서 여러 작업이 있으면 → 여러 `### {HH:MM}` 엔트리로 분리 (각각 자기완결적)

## 3단계: 다음 할 일 제안

- 작업문서의 미완료 todo와 연계하여 다음 단계 구체화
- 성공 → 다음 단계 명령어 / 실패 → 원인 분석 + 재시도 명령어 / 부분 성공 → 실패 부분만 재실행
- 명령어는 VM 환경 규칙에 맞게 조정 (단일 코드블록, 복잡한 파일 전송 시 SCP 분리)

### VM 경로 교차검증 필수

**히스토리 기입·명령어 제안 시 VM 경로를 작성하기 전에 반드시 `{docs_root}/../etc/` (= `personal-work/projects/{project}/etc/`) 내 공통문서에서 실제 경로를 확인하고, 확인된 경로만 사용할 것.**

- 타 프로젝트의 `etc/` 디렉터리 참조 금지 — 해당 프로젝트의 etc/만 사용
- 기존 주제별 문서에 이미 적혀있는 경로도 맹신 금지 — 공통문서와 교차검증 후 불일치하면 수정
- 추측 경로 사용 금지 (`/pipeline/docker/*` 등 임의 경로 절대 금지)
- 공통문서에 해당 환경 경로가 없으면 "경로 미확인"으로 명시하고 빈칸으로 둘 것

## 4단계: 체크박스 갱신

### 자동 갱신 (확인 없이 즉시)

로그에서 **명확히 완료/성공** 확인된 항목 → `[ ]`/`[-]` → `[x]` (주제별 문서 + 인덱스 양쪽)

**인덱스 갱신 필수**: 주제별 문서만 체크하고 인덱스를 안 고치면 carry-over 시 완료 항목이 오이월됨.

### 블로킹 감지 — `[ ]` → `[-]`

로그에서 외부 요인 진행 불가 확인 시 → `[-]`로 변경 + 인덱스 외부 블로커 테이블에 행 추가 (비고에 원본 주제별 문서 링크 필수).

### 블로커 해소 감지

로그에서 기존 블로커 해소 확인 시:
1. 블로커 테이블 상태 갱신 (`**해소**`)
2. 비고의 원본 링크 → 주제별 문서 읽기 → `[-]` 항목 추출
3. 오늘 디렉토리에 주제별 문서 생성 (`[-]` → `[ ]` 변환, `[x]` 제거)
4. 인덱스 작업 목록 + 오늘 할 일에 재등록

### "오늘 할 일"은 건드리지 않음

개별 서브태스크 완료로 "오늘 할 일" 자동 체크 안 함. 블로커 해소 재등록 시만 추가.

### 제안만 (확인 필요)

부분 성공, 에러 포함, 새 todo 항목 → 제안만.

## 5단계: etc/ 고정 문서 자동 갱신

아래 패턴 감지 시 docs-manager Agent를 백그라운드로 호출. 감지 없으면 스킵.

| 감지 패턴 | 갱신 대상 | 동작 |
|---|---|---|
| 방화벽 오픈 확인/실패 | `방화벽_오픈_요청_이력관리.md` | 해당 셀 O/⏳/X 갱신 |
| DB 접속 정보 수신/변경 | `인프라_접속정보.md` | 해당 테이블 갱신 |
| 미래 계획 선행 조건 충족 | `미래_계획.md` | `✓ 충족` 표기 |

## 6단계: updated 타임스탬프 갱신

수정한 문서의 YAML 헤더 `updated` 값을 현재 시각으로 갱신.

# 출력 형식

```
## 히스토리 기입 완료

기입 문서: `{경로}` — `### {HH:MM} — {제목}`
인덱스 갱신: 체크박스 {N}건 (또는 "없음")

## 분석 결과 / 다음 할 일 / 실행 명령어
{내용}

## todo 업데이트
- [x] {완료 항목} ← 자동 갱신
- [-] {블로킹 항목} ← 자동 갱신
- [ ] {제안 항목} ← 추가할까요?
```

# 제약 조건

1. 기존 문서 내용 삭제 금지 — 추가만
2. 로그를 그대로 복사하지 않음 — 분석·요약 정리
3. VM 환경 규칙에 맞게 조정 (단일 코드블록, 복잡한 파일 전송 시 SCP 분리)
4. todo 갱신: 완료 명확 → 자동 갱신 (인덱스+주제별 모두). 불확실 → 제안만
5. CLAUDE.md 제외 규칙 준수 (동료, P3/P4, 블로커 작업 제안 금지)
6. 명령어·경로 제안 전 `{docs_root}/../etc/` 먼저 참조 — 추측 금지
7. `## 히스토리`에는 `### {HH:MM} — {제목}` 엔트리만 생성 — 비타임스탬프 `###` 서브섹션 금지

## --catalog 플래그 (선택)

`--catalog` 플래그가 주어지면 스킬 완료 후 사용자에게 catalog-update 실행 여부를 제안한다.

- 히스토리 처리·문서 기입·todo 갱신이 모두 끝난 뒤 마지막에 아래 안내를 출력한다:
  ```
  catalog-update도 실행할까요? (`/catalog-update --project {현재 프로젝트}`)
  ```
- 실제 자동 실행은 하지 않고 사용자 선택에 맡긴다
- 프로젝트에 `catalog = true` 설정이 없으면 이 제안을 생략한다
