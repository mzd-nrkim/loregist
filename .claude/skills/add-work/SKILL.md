# 역할

오늘(또는 지정일) 작업문서에 새 업무 항목을 추가한다.
인덱스(01.작업문서.md)에는 **체크박스 + 주제문서 링크만** 넣고, 상세 내용은 주제별 문서로 분리 생성한다.

## 프로젝트 해석

1. `--project <key>` 인자가 있으면 그 프로젝트 사용
2. 없으면 `stashdex project current` (cwd 기준 자동 추론)
3. 추론 실패 시 사용자에게 프로젝트 키 질문

`docs_root`: 추론된 프로젝트의 docs_root 값 (`stashdex project list`로 확인)

→ stashdex/CLAUDE.md "스킬 공통 — 프로젝트 추론 규칙" 적용

# 입력 분석

`$ARGUMENTS`를 파싱한다:

1. **주제명** — 첫 번째 인자 (필수)
   - 주제별 문서의 파일명과 인덱스의 `###` 소제목에 사용
   - 예: `DB_스키마_변경_논의자료`, `PRD_Cache_ACL_변경`
   - 공백이 포함된 경우 언더스코어로 변환

2. **상세 내용** — 주제명 뒤의 자유 텍스트 (선택)
   - 주제별 문서의 `## 작업 상세`에 그대로 삽입
   - 마크다운(테이블, 코드블록, 체크박스 등) 지원
   - 인자로 전달하지 않아도, **대화 맥락에서 사용자가 이미 제공한 내용**이 있으면 그것을 사용
   - 예: "오늘 작업문서에 보안등급 논의자료 추가해줘. 내용은 소스A 39테이블과 소스B 52테이블 정리야" → 대화 맥락에서 내용을 추출하여 주제별 문서에 포함

3. **옵션**:
   - `--date YYYY-MM-DD`: 대상 날짜 지정 (기본: 오늘)
   - `--done`: 생성 즉시 완료 처리 (`[x]` + `[완료]`)

인자가 주제명이 아닌 경우(파일 경로 등) 또는 인자가 없는 경우, 대화 맥락·첨부 파일 내용에서 주제를 추론하여 직접 결정한다. 추론이 불가능한 경우에만 사용자에게 물어본다.

## 내용 수집 우선순위

1. `$ARGUMENTS`에 상세 내용이 직접 포함된 경우 → 그대로 사용
2. 대화 맥락에서 사용자가 요청과 함께 내용을 제공한 경우 → 추출하여 사용
3. 기존 파일에서 내용을 가져오라고 지시한 경우 → 해당 파일을 읽어서 정리
4. 위 모두 해당 없으면 → 빈 섹션으로 생성하고 사용자에게 안내

# 경로 규칙

| 구분 | 경로 |
|---|---|
| 인덱스(작업문서) | `{docs_root}/{YYYY-MM-DD}/{YYYY-MM-DD}.01.작업문서.md` |
| 주제별 문서 | `{docs_root}/{YYYY-MM-DD}/{YYYY-MM-DD}.{순번}.{주제명}.md` (순번 02~) |

# 처리 절차

## 1단계: 대상 날짜 디렉토리 확인

- 대상 날짜 디렉토리(`{docs_root}/{YYYY-MM-DD}/`)가 없으면 생성
- 인덱스 파일이 없으면 기본 구조로 생성 (아래 참조)

## 2단계: 다음 순번 결정

- 대상 날짜 디렉토리에서 `{YYYY-MM-DD}.{순번}.*.md` 파일을 glob으로 탐색
- 가장 큰 순번 + 1을 새 순번으로 사용
- 순번은 2자리 zero-padding (예: `02`, `03`, ..., `10`, `11`)

## 3단계: 주제별 문서 생성

아래 구조로 생성한다:

```markdown
---
updated: {YYYY-MM-DD HH:MM}
---
# {주제 제목}

## 컨텍스트

> {사용자가 제공한 배경 설명 또는 빈 상태로 남김}

## 작업 상세

{사용자가 제공한 상세 내용 — 테이블, 코드블록, 체크박스 등}

## 히스토리
```

- 사용자가 대화에서 제공한 내용(테이블, 설명, 코드블록 등)을 `## 작업 상세`에 넣는다
- 내용이 없으면 빈 섹션으로 남긴다

## 4단계: 인덱스에 항목 추가

인덱스의 `## 작업 목록` 섹션 **최하단**에 아래 형식으로 추가한다:

```markdown

### {주제 제목}
> 상세: [{파일명}]({파일명})

- [ ] {체크박스 항목 1}
- [ ] {체크박스 항목 2}
```

### 체크박스 규칙

- 주제별 문서의 `## 작업 상세`에 체크박스가 있으면 → 인덱스에도 **동일한 체크박스**를 넣는다
- 주제별 문서에 체크박스가 없으면 → 주제 전체를 대표하는 체크박스를 **1개 이상** 반드시 생성한다
  - 예: `- [x] 소스A DB + 소스B DB 테이블 목록 및 용도 정리 [완료]`
- **체크박스가 0개인 소주제는 만들지 않는다** — `/daily-report` 스킬이 인식할 수 없음
- `--done` 옵션이면 체크박스를 `[x]`로, 항목 끝에 `[완료]`를 붙인다

### "오늘 할 일" 섹션 연동

- `--done`이 아닌 경우, `## {MM-DD} 오늘 할 일` 섹션에도 해당 항목을 1줄로 요약하여 추가한다
- `--done`인 경우, "오늘 할 일"에는 추가하지 않는다 (이미 완료이므로)
- 항목명은 CLAUDE.md의 **"오늘 할 일" 항목명 네이밍 규칙**을 따른다:
  - 주제명이나 대화 맥락에서 환경/서비스/호스트 정보를 추출하여 접두어로 포함
  - 예: 주제 `ServiceA_QDB_직접조회_테스트` + VM `srv01` → `DEV(srv01) - ServiceA QDB 직접 조회 테스트`
  - `— {상세설명}` 접미사 없이 간결한 동사형으로 마무리

## 5단계: updated 타임스탬프 갱신

- 인덱스와 주제별 문서 모두 YAML 헤더의 `updated` 값을 현재 시각으로 갱신

# 인덱스 신규 생성 시 기본 구조

인덱스 파일이 없을 때 아래 구조로 생성한다:

```markdown
---
updated: {YYYY-MM-DD HH:MM}
---
# {YYYY-MM-DD} 작업문서

## {MM-DD} 오늘 할 일

{항목}

## 외부 블로커

| 블로커 | 상태 | 비고 |
|---|---|---|

## 작업 목록

{소주제 + 체크박스 + 주제문서 링크}
```

# 출력 형식

```
## 업무 추가 완료

- 주제: {주제명}
- 주제별 문서: `{파일 경로}`
- 인덱스 항목: 체크박스 {N}개 추가
- 상태: {미완료 / 완료}
```

# 제약 조건

1. **인덱스에 상세 내용(테이블, 코드블록 등)을 넣지 않는다** — 체크박스 + 링크만
2. **체크박스 0개 금지** — `/daily-report` 인식을 위해 최소 1개 필수
3. **기존 내용 삭제 금지** — 추가/삽입만 수행
4. **순번 충돌 방지** — 기존 파일과 겹치지 않는 순번 사용
5. 코드 변경 없음 — 이 스킬은 작업문서 파일만 수정
6. **"관련 문서" 연결 금지 규칙** — 파생 작업 문서에서 원본 작업의 작업계획서/결과보고서를 **이 작업의 작업계획서**로 연결 금지. 작업 A에서 이슈가 발견되어 작업 B가 파생된 경우, A의 문서는 `발견 경위 참조`로만 표기. 이 작업 자체의 계획서가 없으면 "이 문서가 계획 역할" 또는 "미작성" 명시.
